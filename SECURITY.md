# Security headers

The production static site is hosted on GitHub Pages (`aquarium.vibeai.software`).
GitHub Pages does not support repository-defined arbitrary HTTP response headers,
so browser-enforced page policy is currently limited to metadata that can be
declared inside each HTML document.

## Static pages

`index.html` and `assets/roadmap.html` declare:

- a `Content-Security-Policy` meta tag that limits scripts, styles, images,
  network requests, frames, form posts, objects, workers, and base URIs to the
  origins the current pages need;
- a `referrer` meta tag with `no-referrer`.

The main app permits:

- `https://cdn.jsdelivr.net` for Chart.js;
- `https://gnewkvhgwd.execute-api.eu-west-1.amazonaws.com` for the API;
- `https://*.auth.eu-west-1.amazoncognito.com` for Cognito token exchange and
  form/navigation targets;
- `data:` images for locally uploaded profile avatars.

Both pages still require `'unsafe-inline'` for scripts and styles because the
frontend is a no-build single-file app with inline CSS, inline event handlers,
and inline script blocks. Removing that exception requires moving handlers,
styles, and application JavaScript into external same-origin files or adding a
repeatable nonce/hash generation step.

## API responses

Lambda JSON responses include:

- `Cache-Control: no-store`
- `Referrer-Policy: no-referrer`
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`

These apply to API Gateway responses generated by `lambda/index.mjs`.

## Headers that need a different static host layer

The following protections cannot be fully expressed from GitHub Pages content:

- `Strict-Transport-Security`
- `Permissions-Policy`
- `X-Frame-Options`
- `Cross-Origin-Opener-Policy`
- `Cross-Origin-Resource-Policy`
- CSP `frame-ancestors` and report endpoints

If the static site moves behind CloudFront, Netlify, or another host that can
set response headers, configure those headers at that layer and remove any
duplicated HTML meta fallback only after confirming equivalent browser behavior.
